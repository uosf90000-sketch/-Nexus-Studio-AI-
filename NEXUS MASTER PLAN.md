# ٠. الرؤية — ما الذي نبنيه فعلاً

**المستخدم يدخل فكرة. النظام يعطيه في النهاية رابط تطبيق شغّال منشور.**

السلسلة الكاملة:

```
فكرة
 → مجلس من ثلاثة نماذج يناقشها ويقرر   (المرحلة ٥)
 → Claude Code يبني المشروع كاملاً       (المرحلة ٦)
 → Codex يدقق → حلقة تصحيح               (المرحلة ٦)
 → نشر تلقائي على Railway                (المرحلة ٧)
 → رابط حي يفتحه المستخدم من جواله
```

**الأدوار — كل دور مزوّد مختلف، وهذا مقصود:**

| الدور | المزوّد |
|---|---|
| المدير (يحسم) | ChatGPT |
| المهندس (الجدوى التقنية) | Claude |
| المحلل (السوق والمنافسة) | Gemini |
| المبرمج (يبني) | Claude Code |
| المدقق (يراجع) | Codex |

نموذج يراجع نفسه = ضعف. الفصل بين المُنتج والمدقق أساسي.

**المبادئ الثابتة عبر كل المراحل:**
- كل شيء له سجل: من قرر، ولماذا، وكم كلّف.
- الحدود مفروضة في الكود: تكلفة، محاولات، وقت.
- العزل التام: لا شيء يلمس منصة Nexus أو مفاتيحها.
- المستخدم يقدر يوقف أي شيء، وقرارات الدمج والنشر النهائية له.
- عند أي تعارض أمني أو خطر فقدان بيانات: **أوقف وأبلغ، لا تخمّن.**

# المرحلة ٥ — المجلس

## ٥.١ ما نبنيه

**مجلس من ثلاثة نماذج يناقش الفكرة علناً، ومحضر نقاش يقرأه المستخدم.**

هذه المرحلة **نقاش وقرار فقط** — لا توليد كود.

## ٥.٢ الأعضاء

| العضو | المزوّد | الدور |
|---|---|---|
| **المدير** | ChatGPT (OpenAI) | يؤطّر القرار، يوازن بين الآخرين، يصدر الحكم النهائي |
| **المهندس** | Claude (Anthropic) | الجدوى التقنية، التعقيد، الجهد، المخاطر |
| **المحلل** | Gemini (Google) | السوق، الطلب، المنافسون، التكرار، التمييز |

كل عضو له **adapter خاص** خلف واجهة `AgentAdapter` الحالية. ربط الدور بالمزوّد **في الإعدادات**، لا في الكود.

## ٥.٣ تدفق النقاش

```
المستخدم يدخل فكرة
 → تُفتح جلسة مجلس
 → الجولة ١ (مواقف افتتاحية، بالترتيب):
      المحلل  — السوق، الطلب، من يفعلها الآن، ما الفرق
      المهندس — الجدوى، التعقيد، الجهد، المخاطر التقنية
      المدير  — يؤطّر القرار الحقيقي والأسئلة المفتوحة
 → الجولة ٢ (ردود — كل عضو يرى الجولة ١):
      المحلل والمهندس يردّان على بعضهما وعلى تأطير المدير
 → الحكم:
      المدير يصدر: PROCEED | REVISE | REJECT
      مع سبب مكتوب، وإذا REVISE: ما الذي يجب تغييره تحديداً
```

**الحد: جولتان + حكم.** قابل للتعديل، والافتراضي ٢. كل عضو يستلم المحضر كاملاً حتى لحظته — نقاش حقيقي، لا ثلاثة آراء منفصلة.

## ٥.٤ بروتوكول الحسم

- **الحالة العادية:** المدير يقرر مستنداً لكلام الآخرين، ويجب أن يشير لما قالوه فعلاً.
- **عند الخلاف:** إذا تعارض المحلل والمهندس، **يذكر المدير التعارض صراحة** ويقول أي جهة رجّح ولماذا. ممنوع التجاهل أو المجاملة.
- **عند نقص الأدلة:** الحكم `REVISE` مع تسمية المعلومة الناقصة — لا تخمين.
- **المستخدم يتجاوز الجميع.** الحكم توصية لا بوابة.

## ٥.٥ المحضر مرئي (متطلب أساسي)

كل رسالة تُحفظ وتُعرض: **من** تكلم، **ماذا** قال كاملاً، **أي جولة**، **وكم كلّفت**. تُعرض كمحادثة عربية RTL يقرأها المستخدم كأنه يحضر اجتماعاً — لا JSON، لا ملخص مطوي. وإجمالي تكلفة الجلسة ظاهر دائماً.

## ٥.٦ قاعدة البيانات

- `CouncilSession` — id, projectId, status, verdict, verdictReason, totalCost, createdAt
- `CouncilMessage` — id, sessionId, round, role (DIRECTOR/ENGINEER/ANALYST), provider, model, content, inputTokens, outputTokens, cost, order, createdAt

## ٥.٧ قيود

- إعادة استخدام `AgentAdapter` وآلية إخفاء الأسرار وتتبّع التكلفة.
- **مفاتيح جديدة:** OpenAI و Google — في متغيرات البيئة فقط، لا تُطبع أبداً. مفتاح ناقص = فشل واضح يسمّي العضو العاجز، **بدون** استبدال صامت بنموذج آخر.
- إعادة محاولة بحد ٢ لكل دور. عضو فشل = يُسجّل في المحضر ويكمل المدير بما لديه.
- تحقق Zod من كل مخرج. حكم المدير من القيم الثلاث فقط.
- احترام `COST_WARNING_THRESHOLD`؛ تجاوز الحد يوقف الجلسة.

## ٥.٨ معايير القبول

1. أدخل فكرة وأرى **ثلاثة نماذج مختلفة تتناقش فعلاً** في جولتين ثم حكم.
2. المحضر كامل ومقروء بالعربي RTL مع التكلفة.
3. كل عضو مزوّد مختلف فعلاً — يُتحقق من بيانات المحضر.
4. حكم المدير يشير لكلام الآخرين، والتعارض مذكور صراحة.
5. تكلفة الجلسة ظاهرة والحد يوقف الجلسة الهاربة.
6. البناء والفحص والاختبارات تمر، والمزوّدون كلهم mock.

**المطلوب في التقرير:** محضر نقاش **حقيقي كامل** من فكرة حقيقية — لأحكم بنفسي: هل هذا مجلس فعلاً أم ثلاثة نماذج تتفق بأدب؟

# المرحلة ٦ — المبرمج والمدقق وحلقة التصحيح

> **أخطر مرحلة في المشروع.** أول مرة ينفّذ النظام أوامر فعلية. اقرأ ٦.٢ مرتين.

## ٦.١ ما نبنيه

```
حكم المجلس = PROCEED
 → Nexus ينشئ مساحة عمل معزولة للمشروع
 → المبرمج (Claude Code) يبني المشروع هناك:
      يكتب ملفات، يثبّت حزم، يشغّل اختبارات، يصلّح أخطاءه
 → المدقق (Codex) يراجع المشروع الناتج
 → إذا طلب تغييرات: الملاحظات ترجع للمبرمج → يصلّح → يُدقّق ثانية
 → عند الموافقة (أو بلوغ حد المحاولات): توقف وأبلغ
```

النشر **ليس هنا** — المرحلة ٧.

## ٦.٢ قواعد التنفيذ المطلقة

المبرمج ينفّذ أوامر حقيقية. كل قاعدة تُفرض في الكود:

1. **مساحة عمل معزولة لكل مشروع** — كل شيء داخل `workspaces/<projectId>/`. تحقّق من المسار المطلق قبل كل عملية وارفض أي خروج.
2. **لا يلمس كود Nexus أبداً** — لا مصدره، ولا `prisma/`، ولا `.env*`، ولا قاعدة بياناته. محاولة = إيقاف.
3. **لا وصول لأسرار Nexus** — بيئة المبرمج **خالية** من `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GOOGLE_API_KEY`, `GITHUB_OUTPUT_TOKEN`, `DATABASE_URL`, `ACCESS_CODE`.
4. **حدود على كل شيء** — أقصى جولات، أقصى وقت، أقصى تكلفة. بلوغ الحد = توقف وأبلغ، لا استمرار.
5. **زر إيقاف** — يوقف المهام الجديدة، يحفظ الحالة، لا يحذف بيانات.
6. **سجل تدقيق كامل** — كل أمر، كل ملف، كل دور، كل تكلفة.
7. **لا إجراءات شبكية مدمّرة** — لا نشر، لا GitHub، لا Railway في هذه المرحلة. بناء محلي فقط.

عند تعذّر أي قاعدة: **ارفض وأبلغ.**

## ٦.٣ الوكيلان

**المبرمج — Claude Code (وكيلي):** يُستدعى برمجياً على مساحة العمل. يستلم الفكرة المعتمدة والـ PRD، ولاحقاً ملاحظات المدقق. يبني تكرارياً كما يفعل مع الإنسان.

**المدقق — Codex (OpenAI):** يستلم المشروع الناتج (شجرة الملفات + محتوى الملفات المهمة + نتائج الاختبارات). يرجع تدقيقاً منظّماً:
- `verdict`: APPROVE | REQUEST_CHANGES | REJECT
- `summary` بلغة واضحة
- `issues[]`: `{ severity: HIGH|MEDIUM|LOW, file?, description }`
- `runsCorrectly`: هل يبني ويشتغل فعلاً؟
- `securityConcerns[]`

تحقق Zod. **المدقق مزوّد مختلف عن المبرمج عمداً.**

## ٦.٤ حلقة التصحيح

```
جولة ١: المبرمج يبني → المدقق يدقق
  APPROVE          → انتهى
  REQUEST_CHANGES  → الملاحظات للمبرمج → جولة ٢
  REJECT           → توقف وأبلغ (لا تحرق مالاً)

أقصى جولات: قابل للتعديل، الافتراضي ٣
```

بلوغ الحد بلا موافقة = `NEEDS_HUMAN` مع عرض المشاكل. **لا تدّعِ نجاحاً.** ملاحظات المدقق تمرّ **حرفية**، لا تُلخّص لغموض.

## ٦.٥ قاعدة البيانات

- `BuildRun` — id, projectId, status (RUNNING/APPROVED/NEEDS_HUMAN/STOPPED/FAILED), workspacePath, rounds, totalCost, startedAt, finishedAt
- `BuildRound` — id, buildRunId, roundNumber, builderOutput, auditVerdict, auditIssues (JSON), runsCorrectly, cost, createdAt
- `CommandLog` — id, buildRunId, command, exitCode, truncatedOutput, createdAt

## ٦.٦ الاختبارات المطلوبة (جوهر المرحلة)

بمجلدات مؤقتة، لا المستودع الحقيقي:

1. الكتابة داخل مساحة العمل فقط؛ محاولة الخروج **مرفوضة**.
2. محاولة قراءة/كتابة مصدر Nexus أو `.env*` **مرفوضة**.
3. بيئة المبرمج **خالية** من أسرار Nexus.
4. الحلقة تتوقف عند الحد بـ `NEEDS_HUMAN` — لا حلقة لا نهائية.
5. سقف التكلفة يوقف التشغيل في منتصفه.
6. زر الإيقاف يوقف بناءً جارياً ويحفظ الحالة.
7. `REJECT` من المدقق يوقف الحلقة فوراً.
8. المدقق مزوّد مختلف عن المبرمج — يُتحقق من سجل الجولة.

## ٦.٧ معايير القبول

1. من فكرة معتمدة، يُبنى مشروع حقيقي في مساحة معزولة وأرى ملفاته.
2. Codex يدققه ويرجع ملاحظات منظّمة، وعند REQUEST_CHANGES يصلّح المبرمج فعلاً ويُعاد التدقيق.
3. كل قواعد ٦.٢ مفروضة ومغطاة باختبارات ٦.٦.
4. التكلفة ظاهرة، الحدود مفروضة، الإيقاف يعمل.

**المطلوب في التقرير:** تشغيل حقيقي كامل — شجرة الملفات المنتجة، ملاحظات المدقق الفعلية، عدد الجولات، التكلفة. وأين أهدرت الحلقة مالاً أو دارت بلا تقدّم.

# المرحلة ٧ — النشر وإرجاع الرابط

> هذه المرحلة تحقق وعد المنتج: فكرة → رابط شغّال.

## ٧.١ ما نبنيه

```
BuildRun معتمد من المدقق
 → إنشاء مستودع GitHub جديد لهذا المشروع
 → رفع المشروع المبني إليه
 → إنشاء مشروع + خدمة Railway جديدين مربوطين بالمستودع
 → إضافة قاعدة بيانات إذا احتاجها المشروع
 → ضبط متغيرات بيئة المشروع نفسه
 → إطلاق النشر ومتابعته
 → عند النجاح: حفظ وعرض الرابط الحي
 → عند الفشل: التقاط السجل وعرض الخطأ بوضوح
```

الناتج: **رابط يضغطه المستخدم.**

## ٧.٢ قواعد مطلقة

1. **لا يلمس مستودع Nexus ولا مشروع Nexus على Railway أبداً.** كل تطبيق مولّد له مستودعه ومشروعه المستقلان. تحقّق قبل أي استدعاء وارفض.
2. **لا يُنشر إلا المُدقّق المعتمد.** ممنوع نشر `NEEDS_HUMAN` أو `REJECT` أو غير مكتمل.
3. **الاعتمادات منفصلة.** توكن Railway وتوكن GitHub هنا مقصوران على المشاريع المولّدة — لا مشروع Nexus. في متغيرات البيئة، لا تُطبع، ولا تُسلّم للتطبيق المولّد.
4. **التطبيق المولّد لا يستلم أسرار Nexus أبداً.**
5. **وعي بالتكلفة.** أظهر ما سيُنشأ قبل إنشائه، وسجّل كل مورد حتى لا يبقى شيء يتيم.
6. **سجّل كل شيء** — المستودع، معرّف مشروع Railway، الخدمة، النشر، الرابط، الحالة.

## ٧.٣ التدفق (كل خطوة قابلة للاستئناف)

1. **إنشاء مستودع GitHub** — خاص افتراضياً. اسم موجود = لا تكتب فوقه، اختر اسماً آخر أو اسأل.
2. **رفع المشروع كاملاً** — مع `.gitignore` سليم؛ لا ترفع `.env` ولا أسراراً أبداً.
3. **إنشاء مشروع وخدمة Railway** من المستودع.
4. **تجهيز قاعدة بيانات** إذا احتاجها المشروع، وربط رابطها بمتغيرات الخدمة.
5. **ضبط متغيرات التطبيق المولّد** على Railway.
6. **إطلاق النشر والاستطلاع** حتى النجاح أو الفشل (بمهلة).
7. **عند النجاح:** التقاط الرابط العام وحفظه وعرضه.
8. **عند الفشل:** التقاط سجل البناء وعرض الخطأ الحقيقي — **لا تبلّغ بنجاح**.

كل خطوة idempotent: إعادة نشر فاشل **لا تكرّر** مستودعاً أو مشروعاً أو خدمة.

## ٧.٤ التعامل مع الفشل

- فشل النشر **طبيعي** لا استثنائي. التقط السجل، اعرض الخطأ الحقيقي، علّم `FAILED` بالسبب.
- **إعادة محاولة** تستأنف من الخطوة الفاشلة لا من الصفر.
- اختيارياً: تمرير خطأ النشر للمبرمج كجولة إصلاح — بحد **أقصى محاولتين** ثم للمستخدم.
- **لا تعلن نجاح النشر حتى يستجيب الرابط فعلاً.**

## ٧.٥ قاعدة البيانات

- `Deployment` — id, projectId, buildRunId, githubRepo, railwayProjectId, railwayServiceId, deploymentId, url, status (PENDING/BUILDING/LIVE/FAILED), errorLog, createdAt, updatedAt

## ٧.٦ ما يراه المستخدم

- كل خطوة أثناء حدوثها ("أنشأنا المستودع"، "ننشر الآن…")
- عند النجاح: **الرابط بارزاً** — هذه لحظة تسليم المنتج
- عند الفشل: الخطأ الحقيقي مقروءاً، مع إعادة محاولة
- الموارد المنشأة في حساباته

## ٧.٧ الاختبارات المطلوبة (كلها mock)

1. ينشر بناءً معتمداً: مستودع، مشروع، خدمة، رابط، سجل `Deployment`.
2. **يرفض** نشر بناء غير معتمد.
3. **يرفض** إذا كان الهدف مستودع Nexus أو مشروعه على Railway.
4. إعادة نشر فاشل **لا تكرّر** الموارد.
5. يفشل نظيفاً: `FAILED` والسجل محفوظ، **بلا نجاح كاذب**.
6. لا يعلن `LIVE` حتى يستجيب الرابط.
7. لا يُمرَّر أي سر من أسرار Nexus للتطبيق المولّد.

## ٧.٨ معيار القبول الأهم

> أدخل فكرة، يوافق المجلس، يبني المبرمج، يجيزها المدقق، ويعطيني التطبيق **رابطاً شغّالاً أفتحه من جوالي.**

**المطلوب في التقرير:** ما الذي سينكسر غالباً في أول نشر حقيقي، ولماذا — أفضّل أن أعرف مسبقاً.

# ٨. قواعد عامة لكل المراحل

- معرّفات مهام لكل ملف: `NEXUS-P5-001`, `NEXUS-P6-001`, `NEXUS-P7-001`…

- كل مخرجات النماذج يتحقق منها Zod.
- كل الاختبارات mock — لا استدعاء حقيقي لأي مزوّد أو GitHub أو Railway.
- عند أي تعارض أمني أو خطر فقدان بيانات: **أوقف وأبلغ.**
- **لا تبدأ مرحلة قبل اعتماد المالك للمرحلة السابقة.**
